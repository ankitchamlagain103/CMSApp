# CMSApp

Blog CMS backend built as a Clean Architecture solution on .NET 10, targeting PostgreSQL.

## Solution structure

`CMSApp.slnx` groups four projects under a `/src/` solution folder (virtual grouping only — the projects sit directly under the repo root on disk), referencing each other in one direction:

```
WebApi  -->  Infrastructure  -->  Application  -->  Domain
```

- **Domain** — entities, enums, auditing interfaces, repository *contracts* (`Interfaces/`), and shared domain-level types (`Common/`). No dependency on any other project. `ValueObjects/`, `Aggregates/`, `DomainServices/` exist as empty convention folders for when the domain grows past simple entities — don't force something into them prematurely.
- **Application** — use cases (`<Feature>/Commands`, `<Feature>/Queries`, `<Feature>/Validators`, `<Feature>/Dtos`), the cross-cutting contracts Infrastructure implements (`Common/Interfaces/`), and the shared API response envelope (`Common/Models/`). References `Domain` only.
- **Infrastructure** — EF Core persistence (`Persistence/`, including `Persistence/DataSeeder/`), ASP.NET Core Identity entities/config (`Identity/`, `Identity/EntityConfiguration/`), the `<Feature>Service` implementations that need Identity's managers (`Identity/Services/`), their DTO mappers (`Identity/Mapper/`), and small standalone helpers like the JWT generator (`Common/`). References `Application` (transitively `Domain`).
- **WebApi** — ASP.NET Core Web API host (`Controllers/`, `Middleware/`, `Program.cs`). References `Application` and `Infrastructure`.

Each `.csproj` targets `net10.0` with `<Nullable>disable</Nullable>` and `<ImplicitUsings>enable</ImplicitUsings>`. Keep new projects consistent with that.

Repo-level `Docs/` holds consumer-facing documentation: `UI-Implementation-Guide.md` (the frontend integration reference — response envelope, auth/token lifecycle, every endpoint with request/response shapes; **update it whenever an endpoint's route, request, or response shape changes** — it is the contract a UI team builds against) and `CancellationToken-Guide.md` (how request cancellation flows through the layers and the rules for new code). **Additionally, every new feature/endpoint change gets its own short per-feature guide** — `Docs/<feature>_implementation_guide.md`, lowercase snake_case filename (existing examples: `app_config_implementation_guide.md`, `menu_filter_implementation_guide.md`) — a focused, immediately-implementable page for the UI team: endpoints, request/response JSON, allowed values, failure table, typical UI uses. The per-feature guide complements (never replaces) the corresponding section in `UI-Implementation-Guide.md`; keep both in sync.

## Getting the app running (usage)

1. Set real values in `WebApi/appsettings.json` for anything still a placeholder (`Smtp:*`, `Authentication:Google:ClientId`) — DB connection string, `Jwt:Key`, and `Seed:*` credentials are dev-ready as committed.
2. Create and apply EF Core migrations (**user-owned step** — never done from here). Both the Identity tables and the domain tables live in one `ApplicationDbContext`, so it's a single migration chain.
3. `dotnet run` the `WebApi` project. On startup it runs the three seeders (idempotent, safe on every boot; logged-and-skipped if the DB isn't migrated yet): `IdentitySeeder` (roles + one account per role), `MenuSeeder` (menu/permission catalog + full grant to SuperAdmin), then `AppConfigSeeder` (baseline UI settings). Note the first two are currently commented out in `Program.cs` (a local toggle) — re-enable them for a fresh database.
4. Log in as the seeded superadmin (`Seed:SuperAdmin` credentials) — that account works end-to-end immediately, including every protected endpoint. Use it to create menus/roles/claims for everyone else. Swagger UI is at `/swagger` in Development.

**Database provider is PostgreSQL** (`Npgsql.EntityFrameworkCore.PostgreSQL`), not SQL Server — `ConnectionStrings:DefaultConnection` in `WebApi/appsettings.json` is a Postgres connection string. **Migrations are owned by the user, not created here** — don't run `dotnet ef migrations add`/`database update` or otherwise touch migration files unless explicitly asked.

## Domain layer conventions

- Auditing is modeled through interfaces in `Domain/Entities/Interfaces/`: `IAuditableEntity` (`CreatedBy`/`CreatedTs`/`UpdatedBy`/`UpdatedTs`) and `ISoftDeleteAuditableEntity : IAuditableEntity` (adds `IsDeleted`/`DeletedBy`/`DeletedTs`). `AuditableEntity`/`SoftDeleteAuditableEntity` are the plain-class implementations that domain entities inherit directly (`Menu : SoftDeleteAuditableEntity`). ASP.NET Core Identity entities implement the interface directly instead (can't multiple-inherit from `IdentityUser<T>`).
- **Audit stamping is automatic** — `ApplicationDbContext.SaveChangesAsync` stamps `CreatedBy`/`CreatedTs`/`UpdatedBy`/`UpdatedTs` on every tracked `IAuditableEntity`, and converts a `Remove()` on an `ISoftDeleteAuditableEntity` into a `Modified` update that sets `IsDeleted = true` + `DeletedBy`/`DeletedTs`, instead of an actual `DELETE`. **Never set these audit fields manually in a handler or repository** — the `DbContext` does it from `ICurrentUserService.UserName` (falls back to `"system"` when there's no authenticated caller).
- `Domain/Interfaces/IRepository<TEntity, TKey>` is the generic repository contract (`GetByIdAsync`, `GetAllAsync`, `GetPagedAsync` → `Domain/Common/PagedResult<TEntity>`, `AddAsync`, `Update`, `Remove`). Entity-specific repositories extend it (`IMenuRepository : IRepository<Menu, int>`) and add only the queries that don't fit the generic shape (`CodeExistsAsync`).
- Enums live in `Domain/Enums/` as plain `enum` types, stored by EF as their underlying `int`.
- Identity primary key type is `Guid` throughout — keep new identity-related types on `Guid`.

## Identity model (`Infrastructure/Identity/`)

| Entity | Base | Extra state |
|---|---|---|
| `ApplicationUser` | `IdentityUser<Guid>`, `ISoftDeleteAuditableEntity` | profile fields (`FirstName`, `Gender`, `UserType`, `Dob` — nullable, ...), per-user IP allowlist (`IsIpRestricted` + comma-separated `UserIpAllowed` — managed via `PUT /api/users/{id}`, **enforced as of 2026-07-12** at token issuance (login/Google/refresh) and on every authenticated request in `AuthorizedAction`, via the shared `Infrastructure/Common/IpAllowlistChecker`; see `Docs/user_ip_restriction_implementation_guide.md`), nav collections to tokens/roles/logins |
| `ApplicationRole` | `IdentityRole<Guid>`, `IAuditableEntity` | `Description`, nav collections to user-roles/role-claims |
| `ApplicationUserRole` | `IdentityUserRole<Guid>`, `IAuditableEntity` | nav to `ApplicationUser`/`ApplicationRole` |
| `ApplicationUserClaim` | `IdentityUserClaim<Guid>`, `IAuditableEntity` | `MenuId`/`Menu` — claims are menu/permission-scoped, not free-form |
| `ApplicationRoleClaim` | `IdentityRoleClaim<Guid>` | `MenuId`/`Menu`, nav to `ApplicationRole` — **not** auditable |
| `ApplicationUserLogin` | `IdentityUserLogin<Guid>`, `IAuditableEntity` | nav to `ApplicationUser` |
| `ApplicationUserToken` | `IdentityUserToken<Guid>` | `RefreshTokenExpiryDate`, `TokenRefreshTokenRevokedDate`, computed `IsExpired`/`IsRevoked` — **not** auditable. **These custom fields are no longer used** — refresh tokens now live in the dedicated `RefreshToken` entity below; this type still exists only because Identity's own `AddEntityFrameworkStores` needs *a* `TUserToken` type for its internal token storage. Note the table is currently always **empty**: email-confirmation/password-reset tokens are stateless `DataProtectorTokenProvider` tokens (validated cryptographically, never persisted); rows would only appear if MFA is added (authenticator key + recovery codes are stored here by `UserManager`) or if `SetAuthenticationTokenAsync` is used to store external-provider tokens |
| `RefreshToken` | `AuditableEntity` | **Not** an ASP.NET Core Identity type — a project-specific entity (`Id`, `UserId`, `Token`, `ExpiresAtUtc`, `RevokedAtUtc`, `ReplacedByToken`, computed `IsExpired`/`IsRevoked`/`IsActive`, nav to `ApplicationUser`). One row per issued token (not one row per user), which is what makes multi-device sessions possible — see Refresh tokens section below |

**Why `RefreshToken` lives in `Infrastructure/Identity/` and not `Domain/Entities/`:** it holds a navigation property to `ApplicationUser`, which is an Infrastructure type (it inherits `IdentityUser<Guid>` from an ASP.NET Core package Domain doesn't reference) — putting `RefreshToken` in Domain would force a Domain→Infrastructure reference, which inverts the dependency direction. More fundamentally, a refresh token is an *authentication session artifact*, not a business concept: nothing in the blog/CMS domain knows or cares about token rotation. The rule generalizes: an entity belongs in `Domain/Entities/` only if it's a business concept with no compile-time ties to Identity types; anything that references `ApplicationUser`/`ApplicationRole` directly must live in `Infrastructure/Identity/` beside them (it can still reuse Domain's `AuditableEntity` base — Infrastructure→Domain is the allowed direction).

The `Menu` entity (`Domain/Entities/Menu.cs`) doubles as the permission catalog: a claim's `MenuId` points at the menu/permission node it grants, instead of Identity's default free-text `ClaimType`/`ClaimValue` alone.

### Menu hierarchy (`MenuType`/`MenuFor`)

`Menu.MenuType`/`Menu.MenuFor` are plain `string` columns (not a C# `enum` — the DB column is untyped `string`, so changing this to a real enum would need a migration, which is off-limits here), but their allowed values are pinned down as constants rather than free-form: `Domain/Constants/MenuTypes.cs` (`MainMenu = "MAIN_MENU"`, `SubMenu = "SUB_MENU"`, `Permission = "PERMISSION"`, plus an `All` array) and `Domain/Constants/MenuAudience.cs` (`Admin`/`User`/`Both`, same shape). Always construct/compare against these constants, never inline the string literals.

`Menu` is a self-referencing tree via `ParentId`/`MainMenu`/`Childrens` (see `MenuConfiguration`'s self-referencing FK note above), and the three `MenuType` values are the tree's levels:

- `MAIN_MENU` — top-level, **must not** have a parent.
- `SUB_MENU` — **must** have a parent, and that parent must be a `MAIN_MENU`.
- `PERMISSION` — the leaf that `AuthorizedAction` actually checks against (`Menu.Controller`/`Menu.Action`) — **must** have a parent, and that parent may be **either** a `MAIN_MENU` or a `SUB_MENU`. Both `MAIN_MENU → SUB_MENU → PERMISSION` and the shorter `MAIN_MENU → PERMISSION` (skipping the sub-menu level) are valid.

`MenuService.CreateMenuAsync` enforces exactly this via a private `ValidateHierarchyAsync` (fetches the parent through `IUnitOfWork.Menus.GetByIdAsync` and checks its `MenuType`), returning `ResponseCodes.ValidationError` on violation — same pattern as the rest of the validation pipeline (validate → check invariants → mutate). `CreateMenuCommandValidator` separately restricts `MenuType`/`MenuFor` to `MenuTypes.All`/`MenuAudience.All` (the usual `NotEmpty()` in one `RuleFor`, `Must(...)` guarded by `.When(!string.IsNullOrEmpty(...))` in a second `RuleFor` — see the note below about why `.NotEmpty()` and a null-guarded `.Must()` can't share one chain). `UpdateMenuAsync` runs the same `ValidateHierarchyAsync` (now taking `(menuType, parentId)` primitives so create and update share it) plus a self-parent guard and a code-uniqueness re-check only when `Code` changes. `DeleteMenuAsync` is a **soft** delete (`Menu` is `SoftDeleteAuditableEntity`) refused with `Conflict` while the menu still has children — a hidden parent would orphan its visible children in every tree build (`IMenuRepository.HasChildrenAsync`). Corollary of soft delete: a deleted menu's `Code` stays reserved — the unique DB index on `code` still sees the soft-deleted row, so `CodeExistsAsync` uses `IgnoreQueryFilters()` (fixed 2026-07-12; it was previously subject to the `!IsDeleted` filter, letting the service check pass and the insert die at the database as a 500) and recreating a deleted code now fails cleanly with `Conflict` ("is already in use (possibly by a soft-deleted menu)").

**Heads up when editing `DefaultEnabledMenu`/`CriticalChanges` keys**: ASP.NET Core's default route derives the controller-name route value by stripping the `Controller` suffix and keeping the rest as-is (no singularization) — `MenusController` → `"Menus"`, `AuthController` → `"Auth"` — and `AuthorizedAction` compares against that raw route value, so a key that doesn't exactly match it silently never applies. This bit once: the section shipped with dead keys (`"Account"`, `"Role"`, `"Menu"`, `"Common"`) that matched no controller, which meant every non-SuperAdmin user got a 403 on `auth/logout` and `auth/change-password` until the keys were fixed (2026-07-10) to `"Auth": "Logout,ChangePassword,SetPassword"`, `"Roles": "GetUserRoles"`, `"Configs": "GetConfigsByTypeCode"`. When adding an entry, verify both halves against the real controller class and action method names.

**Any property backing a value type must actually be nullable if the EF config marks it optional** — `ApplicationUser.Dob` was originally a non-nullable `DateTime` while its config called `.IsRequired(false)`, which throws `InvalidOperationException` at model-build time (`DateTime` can't be "nullable/optional" without being `DateTime?`). Fixed by making the property `DateTime?`. Watch for this whenever a config calls `.IsRequired(false)` on a value-typed property.

## EF Core entity configuration pattern

Two shared generic base configurations in `Infrastructure/Persistence/EntityConfigurations/` do the auditing column mapping so it isn't repeated per entity: `AuditableEntityConfiguration<T>` and `SoftDeleteAuditableEntityConfiguration<T>` (adds `is_deleted`/`deleted_ts`/`deleted_by`, a global query filter `!IsDeleted`, and an index on `is_deleted`).

Concrete configurations live next to the entities they configure (`Infrastructure/Identity/EntityConfiguration/` for identity types, `Infrastructure/Persistence/EntityConfigurations/` for domain types) and follow this recipe:

1. Inherit the matching generic base **only if the entity implements the matching interface**; otherwise implement `IEntityTypeConfiguration<TEntity>` directly (`ApplicationRoleClaim`, `ApplicationUserToken`).
2. `override void Configure(...)` starts with `base.Configure(builder);` when there is a base to call.
3. `builder.ToTable("snake_case_plural_name", schema: "identity")` for identity tables; **no schema argument** for domain tables (`MenuConfiguration` has none).
4. Every mapped property gets an explicit `.HasColumnName("snake_case")`.
5. Composite keys for Identity join/link tables mirror ASP.NET Core Identity's own defaults: `ApplicationUserRole` on `(UserId, RoleId)`, `ApplicationUserLogin` on `(LoginProvider, ProviderKey)`, `ApplicationUserToken` on `(UserId, LoginProvider, Name)`.
6. Relationships are configured from the "many"/child side only — never duplicated from the parent side. **Always write the `HasOne(...).WithMany(...).HasForeignKey(...)` explicitly when the FK property name doesn't literally match the navigation name + `Id`** — e.g. `RefreshToken.UserId` with a nav named `ApplicationUser`: EF's convention only looks for `ApplicationUserId`, so without the explicit mapping it silently creates a *shadow* `ApplicationUserId` FK column and leaves `user_id` as unrelated data (`RefreshTokenConfiguration` documents this in a comment).
7. `LoginProvider`/`ProviderKey`/`Name` use `HasMaxLength(128)` (Identity's own default); other strings without a business rule are left unbounded.
8. Unique/lookup indexes get an explicit `HasDatabaseName("ix_<table>_<column>")`.
9. Self-referencing hierarchies (`Menu.ParentId` → `MainMenu`/`Childrens`) use `.OnDelete(DeleteBehavior.Restrict)` to avoid multiple-cascade-path errors.

New entity → add the entity + a sibling `*Configuration.cs`. Nothing else needs registering — see DbContext section below.

## DbContext (`Infrastructure/Persistence/ApplicationDbContext.cs`)

`ApplicationDbContext : IdentityDbContext<ApplicationUser, ApplicationRole, Guid, ApplicationUserClaim, ApplicationUserRole, ApplicationUserLogin, ApplicationRoleClaim, ApplicationUserToken>` — **one combined context** for both Identity and domain aggregates (deliberately not split into a separate Identity context). Constructor takes `ICurrentUserService` for audit stamping (see above). `OnModelCreating` calls `base.OnModelCreating` then `builder.ApplyConfigurationsFromAssembly(Assembly.GetExecutingAssembly())` — every `IEntityTypeConfiguration<T>` in the `Infrastructure` assembly is picked up automatically, no manual registration.

Non-Identity `DbSet`s: `public DbSet<Menu> Menus => Set<Menu>();`.

## Application layer: response envelope (`Application/Common/Models/`)

**Every API response is wrapped — controllers never return a bare DTO, an anonymous object, or `dynamic`.**

- `CommonResponse<T>` — `ResponseCode`, `ResponseMessage`, `Data`. Build via `CommonResponse<T>.Success(data, message)` / `CommonResponse<T>.Fail(responseCode, message)`, never `new CommonResponse<T> { ... }` directly outside those factory methods.
- `ResponseCodes` — the string constants passed to `.Fail(...)` (`ValidationError`, `NotFound`, `Conflict`, `Unauthorized`, `Forbidden`, `TooManyRequests`, `ServerError`); add new codes here rather than inlining string literals.
- `PaginatedResponse<T>` — `Items`, `Page`, `PageSize`, `TotalCount`, plus computed `TotalPages`/`HasPreviousPage`/`HasNextPage`. List endpoints return `CommonResponse<PaginatedResponse<TDto>>`.
- **Before every `return` in a handler or controller, build a named local (`var response = ...;` / `var menuDto = ...;`) and return that** — don't construct the return value inline in the `return` statement. This applies to the response wrapper *and* to whatever `Data` holds: define a real class for it (a DTO under `<Feature>/Dtos/`), never an anonymous type.

`Application/Common/Interfaces/` holds the contracts Infrastructure implements: `IUnitOfWork` (`Menus` repository + `SaveChangesAsync`), `ICurrentUserService` (`UserId`/`UserName`/`IsAuthenticated`, read from `HttpContext.User` claims), `IIdentityService` (`GetUserNameAsync`, `IsInRoleAsync`, `CreateUserAsync` → `IdentityOperationResult`, `DeleteUserAsync`). `IIdentityService.CreateUserAsync` is a **minimal low-level primitive** — as of 2026-07-12 it takes username/email/password/firstName/lastName (the two name fields are `NOT NULL` in the DB, so creates through it now succeed), hardcodes `UserType.User`, and still doesn't cover roles/ToS/profile fields — `IUserService.CreateUserAsync` remains the full-featured path.

## Application layer: feature pattern (`Application/<Feature>/`)

**No MediatR, no per-operation handler classes.** Each feature is one `I<Feature>Service` / `<Feature>Service` pair — a single service class with one method per use case, injected by constructor and called directly from the controller. Commands/Queries still exist as plain input classes for operations with more than one parameter, but they're method *parameters*, not the trigger for a dedicated handler class.

`Application/Menus/` is the reference implementation, copy its shape for new features:

```
Menus/
├── Commands/
│   └── CreateMenuCommand.cs      # plain DTO-shaped input (multi-field create/update payloads)
├── Queries/
│   └── GetMenusQuery.cs          # plain DTO-shaped input (Page/PageSize, filters, ...)
├── Validators/
│   └── CreateMenuCommandValidator.cs   # FluentValidation AbstractValidator<TCommand>
├── Dtos/
│   └── MenuDto.cs                # outward-facing shape, never return the entity itself
├── MenuMapper.cs                 # static ToDto(entity) -> Dto, shared by every method on the service
├── IMenuService.cs               # CreateMenuAsync(command), GetMenuByIdAsync(id), GetMenusAsync(query)
└── MenuService.cs                # the one implementation, injected wherever the feature is needed
```

A single-field lookup (`GetMenuByIdQuery` for just an `Id`) doesn't get its own class — take the primitive (`int id`) directly as a method parameter instead. Only give an operation a dedicated Command/Query class once it has enough fields that a bare parameter list would be unwieldy (`CreateMenuCommand`, `GetMenusQuery`).

`MenuService` method responsibilities, in order: validate (`_createMenuCommandValidator.Validate(command)`, fail with `ResponseCodes.ValidationError` and every `ValidationResult.Errors` message joined via a private `BuildValidationErrorMessage` helper), check domain invariants via the repository (e.g. `CodeExistsAsync` → `ResponseCodes.Conflict`), mutate through `IUnitOfWork` (`AddAsync` + `SaveChangesAsync` — never call `DbContext` directly from a service), map to DTO via the feature's mapper, wrap in `CommonResponse<T>.Success(...)`.

Registration in `Application/DependencyInjection.cs` (`AddApplication()`) is one `services.AddScoped<IMenuService, MenuService>();` line per feature — validators are still picked up automatically via `AddValidatorsFromAssembly`, but the service interface/implementation pair needs its own explicit registration (no assembly scanning for services).

**Where the `<Feature>Service` implementation lives depends on what it's built on.** `MenuService` lives entirely in `Application/Menus/` because it's built on `IUnitOfWork`/`IMenuRepository` — abstractions Application already owns. `Application/Users/`, `Application/Roles/`, and `Application/Auth/` only hold the *contracts* (`IUserService`/`IRoleService`/`IAuthService`, their Commands/Queries/Validators/Dtos) — the implementations (`UserService`, `RoleService`, `AuthService`) live in `Infrastructure/Identity/Services/`, with their DTO mappers (`UserMapper`, `RoleMapper`, `RoleClaimMapper`) in `Infrastructure/Identity/Mapper/`, because they're built directly on `UserManager<ApplicationUser>`/`RoleManager<ApplicationRole>`/`SignInManager<ApplicationUser>`, and those Identity types are Infrastructure-only — Application can't reference them. Registration for these still happens the same way, just in `Infrastructure/DependencyInjection.cs` instead of `Application/DependencyInjection.cs` (alongside `ICurrentUserService`/`IIdentityService`). Rule of thumb: if the feature's persistence goes through the generic repository/`IUnitOfWork`, the service belongs in `Application`; if it goes through ASP.NET Core Identity's managers, the service belongs in `Infrastructure/Identity/Services/`.

## Coding style

- 4-space indentation, one class per file, file-scoped `namespace X { }` block (braces, not `namespace X;`) matching the folder path.
- Public auto-properties, no backing fields, `= new List<T>();` default for nav/collection properties.
- **No expression-bodied methods and no ad-hoc lambda-as-logic** in handlers/services/controllers — write full block-bodied methods with explicit `return` statements, and prefer `foreach` over `.Select()`/`.Where()` chains when building a result list (see `GetMenusQueryHandler` mapping `pagedMenus.Items` into DTOs via `foreach`, not LINQ). Framework-required delegate parameters (`AddDbContext<T>(options => ...)`, FluentValidation's `RuleFor(x => x.Prop)`, EF's `Where(x => x.Id == id)`) are the unavoidable exception — keep those to a single simple expression, don't chain further logic inside them.
- Object construction: object-initializer syntax is fine and preferred (`var entity = new Menu { Code = ..., DisplayName = ... };`), including for the response envelope's `Data` payload — just make sure the payload's type is a real class, not an anonymous type.
- Fluent EF configuration calls are chained one member per line, aligned under the receiver.
- Nullable reference types are disabled project-wide — don't add `?` on reference types or assume NRT warnings apply (but *do* make value-type properties nullable when the business rule is "optional", per the `Dob` note above).
- **CancellationToken everywhere** (full rationale + flow diagram in `Docs/CancellationToken-Guide.md`): every async interface method ends with `CancellationToken cancellationToken = default`; every controller action declares `CancellationToken cancellationToken` and forwards it; implementations pass it to every EF Core/SMTP call that accepts one; private async helpers take it as a *required* parameter. Identity's `UserManager`/`RoleManager`/`SignInManager` calls can't take one (no such overloads) — that's expected, not a bug. **Security cleanup after an irreversible step runs with `CancellationToken.None`** (revoking sessions after a password change, retiring the old token after refresh rotation) — a client disconnect must not be able to skip those. `ExceptionHandlingMiddleware` treats `OperationCanceledException` from a genuinely aborted request as an info-level non-event, not a 500.

## Infrastructure: repositories, unit of work, identity services

- `Infrastructure/Persistence/Repositories/Repository<TEntity, TKey>` — generic `IRepository<TEntity, TKey>` implementation over `ApplicationDbContext.Set<TEntity>()`. Entity-specific repositories (`MenuRepository`) inherit it and add only their extra query methods.
- `Infrastructure/Persistence/UnitOfWork.cs` — implements `IUnitOfWork`, lazily constructs repositories (`Menus` property, explicit null-check + assign, not `??=`), and delegates `SaveChangesAsync` to the `DbContext`. Add one property per aggregate repository as new features are built — don't add a generic `Repository<T>` escape hatch to `IUnitOfWork`.
- `Infrastructure/Identity/Services/CurrentUserService.cs` — implements `ICurrentUserService` off `IHttpContextAccessor`. `UserId` reads `ClaimTypes.NameIdentifier` first, then raw `sub` as a fallback — the JWT bearer middleware's default inbound claim mapping renames the token's `sub` claim to `ClaimTypes.NameIdentifier` before it reaches `HttpContext.User`, so reading only the raw JWT name silently returns null on every request (this exact bug once made `AuthorizedAction` 403 everything). `UserName` reads the custom `userName` claim, which the mapping leaves untouched. Requires `<FrameworkReference Include="Microsoft.AspNetCore.App" />` in `Infrastructure.csproj` (a plain class library needs that to see `IHttpContextAccessor`/`Microsoft.AspNetCore.Http` types). JWT bearer auth is now configured (see Authentication & Authorization below), so these claims are populated for any request carrying a valid token.
- `Infrastructure/Identity/Services/IdentityService.cs` — implements `IIdentityService` off `UserManager<ApplicationUser>`.
- `Infrastructure/DependencyInjection.cs` (`AddInfrastructure(services, configuration)`) — registers the `DbContext` (`UseNpgsql`, connection string from `ConnectionStrings:DefaultConnection`), `AddIdentityCore<ApplicationUser>().AddRoles<ApplicationRole>().AddEntityFrameworkStores<ApplicationDbContext>()`, `IHttpContextAccessor`, and the `ICurrentUserService`/`IIdentityService`/`IUnitOfWork` implementations. Called from `WebApi/Program.cs` alongside `AddApplication()`.

## WebApi

- `Controllers/` — one controller per aggregate/feature (`MenusController`), constructor-injects the feature's `I<Feature>Service` directly (no base controller, no mediator). Every action returns `ActionResult<CommonResponse<T>>`; branch on `response.ResponseCode` to pick `Ok`/`BadRequest`/`NotFound` — the service already decided success/failure, the controller just picks the HTTP status.
- `Middleware/ExceptionHandlingMiddleware.cs` — catches unhandled exceptions, logs them, and writes a `CommonResponse<object>.Fail(ResponseCodes.ServerError, ...)` JSON body with a 500 status, instead of leaking the default ASP.NET Core error page/stack trace. Registered first in the pipeline (`app.UseMiddleware<ExceptionHandlingMiddleware>();`, before `UseOpenApi`/`UseHttpsRedirection`) so it wraps everything, including DI activation failures inside a controller.
- `Middleware/SecurityHeadersMiddleware.cs` (2026-07-12) — stamps `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer` on every response; registered right after the exception middleware. Deliberately no CSP (would break Swagger UI, and a JSON API serves no other HTML). `UseHsts()` runs outside Development.
- **CORS is an explicit allowlist** (2026-07-12) — `Program.cs` reads `App:AllowedOrigins` (defaults: `http://localhost:3000`, `http://localhost:5173`) and only calls `UseCors` when the list is non-empty; an empty list means no cross-origin access (fail-closed — never reintroduce `SetIsOriginAllowed(origin => true)` alongside `AllowCredentials()`). Per-environment frontend origins go in that config section. See `Docs/security_hardening_implementation_guide.md`.
- `Program.cs` calls `builder.Services.AddApplication()` and `builder.Services.AddInfrastructure(builder.Configuration)` before `builder.Build()`.
- OpenAPI/Swagger is done via **NSwag** (`NSwag.AspNetCore`, `AddOpenApiDocument` / `UseOpenApi` / `UseSwaggerUi`) — deliberately not `Microsoft.AspNetCore.OpenApi`/`Microsoft.OpenApi` or Swashbuckle, switched because of a vulnerable/deprecated `Microsoft.OpenApi` transitive version. Don't reintroduce those packages. `AddOpenApiDocument` also registers a `"JWT"` API-key security scheme (`Authorization` header) with `AspNetCoreOperationSecurityScopeProcessor` so Swagger UI's "Authorize" button works — paste `Bearer {token}`.

## Authentication & authorization

The user-management implementation follows two written guides the user provided (`UserManagement-Implementation.md` and `ASP.NET-Core-Identity-Implementation-Guide.md`) fairly closely; deviations from them are called out explicitly below since they were deliberate engineering judgment calls, not oversights.

- **JWT bearer auth** is configured in `Infrastructure/DependencyInjection.cs` (`RegisterAuthentication`): `AddAuthentication(JwtBearerDefaults.AuthenticationScheme).AddJwtBearer(...)` reads `Jwt:Key`/`Jwt:Issuer`/`Jwt:Audience` from configuration and validates issuer, audience, lifetime, and signing key (`ClockSkew = TimeSpan.Zero`). `WebApi/Program.cs` calls `app.UseAuthentication()` before `app.UseAuthorization()`.
- `appsettings.json` has a `Jwt` section (`Key`/`Issuer`/`Audience`/`ExpiryMinutes`/`RefreshTokenExpiryDays`). **`Jwt:Key` is currently the literal placeholder `"REPLACE_WITH_A_GENERATED_SECRET_AT_LEAST_32_CHARS_LONG"` — not a real secret.** Generate a real random 32+ byte key before this runs against anything that matters, same caution that already applies to the DB connection string in that file.
- **`WebApi/Filters/AuthorizedAction.cs` is registered globally** in `Program.cs` (`AddControllers(options => options.Filters.Add<AuthorizedAction>())`) — every controller action requires it unless the action or controller has `[AllowAnonymous]`. In order, it:
  1. Passes straight through if `[AllowAnonymous]` is on the endpoint's metadata.
  2. Resolves `ICurrentUserService` **per-request via `context.HttpContext.RequestServices`, not constructor injection** — attribute-derived filters can be instantiated once and reused across requests, so nothing scoped can safely be a constructor parameter; only `IConfiguration` (a singleton) is. 403s immediately (`CommonResponse<object>.Fail(ResponseCodes.Forbidden, ...)`) if there's no authenticated user.
  3. Loads the **full user row from the database** (a request-scoped `ApplicationDbContext` via the same `RequestServices` pattern, `AsNoTracking` lookup on `Users` by the id from `ICurrentUserService.UserId`) — deliberately *not* trusting JWT claims, so a demotion, soft delete, or allowlist change takes effect on the very next request instead of waiting out the token's lifetime (the global `!IsDeleted` query filter means a soft-deleted user simply isn't found → 403). If the lookup finds no user → 403.
  3½. **Per-user IP restriction** (2026-07-12): if the row has `IsIpRestricted = true`, the request's `Connection.RemoteIpAddress` must match the user's `UserIpAllowed` list (`Infrastructure/Common/IpAllowlistChecker` — exact-address match, IPv4-mapped-IPv6 normalized, loopback-equivalent, fail-closed on empty list) or it 403s — this runs **before** the SuperAdmin bypass, so a restricted SuperAdmin is held to their allowlist too.
  3¾. **SuperAdmin bypass**: if `UserType.SuperAdmin` → skips the permission lookup entirely and lets the request through. (`ICurrentUserService` no longer exposes a `UserType` property at all; the `userType` claim is still stamped into tokens by `JwtTokenGenerator` for client-side use, but nothing server-side trusts it.) SuperAdmin therefore doesn't *need* role-claim rows to call anything; `MenuSeeder` still grants them all for the sake of menu-tree rendering and consistency.
  4. Checks the `DefaultEnabledMenu` config section — shape is `{ "DefaultEnabledMenu": { "SomeController": "Action1,Action2" } }` — and lets any authenticated user through for a listed controller/action pair with no permission-row lookup.
  4½. **Critical-change access logging**: on every *authorized* path (SuperAdmin bypass, `DefaultEnabledMenu`, and permission-check pass alike), if the controller/action pair is listed in the `CriticalChanges` config section (same shape as `DefaultEnabledMenu`), a `SystemAccessLog` row is written before the action runs — see System access log section below.
  5. Otherwise reuses that `ApplicationDbContext` and checks `UserRoles` → `RoleClaims` → `Menus` for a row where `Menu.Controller`/`Menu.Action` match the current route, passing `HttpContext.RequestAborted` to the queries. Its broad `catch` (→ "not authorized") deliberately re-throws `OperationCanceledException` first so a client disconnect isn't misreported as a 403.
- **`[AllowAnonymous]` marks the auth endpoints only** (as of 2026-07-12 — there is **no public self-registration**; this is an admin-managed backend): `login`, `google`, `refresh-token`, `verify-email`, `resend-verification-email`, `forgot-password`, `reset-password`. Everything else — all of `Menus`, `Roles`, `Users` (including create, gated by the `USER_CREATE` permission), and `auth/logout`/`auth/change-password`/`auth/set-password` (those three pass via the `DefaultEnabledMenu` `"Auth"` entry, no permission row needed) — needs a valid JWT *and* (for non-`Auth` controllers) a matching `Menus`/`ApplicationRoleClaim`/`ApplicationUserRole` row. `MenuSeeder` seeds that permission data for the **SuperAdmin role only** (see Menu & permission seeding below); users in other roles 403 until claims are granted to their roles via the API.
- **Rate limiting (2026-07-12)**: `AuthController` carries `[EnableRateLimiting("Auth")]` — a fixed-window limiter partitioned by client IP (`WebApi/Extensions/RateLimitingExtensions.cs`, defaults 10 req/60s, configurable via the `RateLimiting` appsettings section; `app.UseRateLimiter()` sits before `UseAuthentication` in `Program.cs`). Rejections return HTTP 429 with the standard envelope (`ResponseCodes.TooManyRequests`). Non-auth endpoints are deliberately unlimited for now; the same caveat as the IP allowlist applies behind a reverse proxy (all callers share the proxy's IP without `ForwardedHeadersMiddleware`).

### User creation (`POST /api/users` — **admin-gated as of 2026-07-12**, no longer `[AllowAnonymous]`)

`Application/Users/Commands/CreateUserCommand.cs` + `Validators/CreateUserCommandValidator.cs` + `Infrastructure/Identity/Services/UserService.CreateUserAsync`:

- Email/username trimmed before uniqueness checks and persistence.
- Password policy (`Application/Common/Validation/PasswordRules.cs`, shared with `ChangePasswordCommandValidator` and `ResetPasswordCommandValidator`): 8–128 chars, upper+lower+digit+special character required, rejected if it equals the email/username (case-insensitive) or appears in a small hardcoded common-password denylist.
- `FirstName`/`LastName` required and trimmed; `FirstName`/`MiddleName`/`LastName` all reject literal `<`/`>` characters as a basic stored-XSS mitigation (**not** full HTML sanitization — if these fields ever get rendered unescaped anywhere, that's a separate concern this doesn't fully cover).
- `Dob` optional; if present, age must be 13–120 (`Application/Users/Validators/UserValidationRules.cs`, shared with `UpdateUserCommandValidator`).
- `PhoneNumber` optional; if present, must be E.164 (`^\+[1-9]\d{1,14}$`).
- `CountryIso3` optional; if present, must be 3 uppercase letters — **shape-only, not a real ISO-3166 whitelist** (the guide calls for "supported country validation" but this project has no canonical country list to validate against).
- `IsTosAgreed` must be `true`.
- `UserType` is **not a field on `CreateUserCommand` at all** — `UserService.CreateUserAsync` hardcodes `UserType.User`. Promoting a user to `Admin`/`SuperAdmin` only happens via `PUT /api/users/{id}` (`UpdateUserCommand` still has `UserType`), and **changing `UserType` there requires the caller to be a SuperAdmin** (added 2026-07-12: `UpdateUserAsync` compares the command's `UserType` to the target's current one; if it differs, a private `IsCallerSuperAdminAsync` loads the *caller's* row from the DB via `ICurrentUserService.UserId` — DB, not token claim, same reasoning as `AuthorizedAction` — and non-SuperAdmins get `Forbidden`). Holding the `USER_UPDATE` permission alone no longer allows promotion; sending the unchanged `UserType` value is always fine.
- On success: sets `LastPasswordChangedTs = UtcNow`, assigns roles, sends a verification email (see below), and returns "Please check your email to verify your account." Role assignment: `CreateUserCommand` has an optional `List<Guid> RoleIds` — when non-empty, the user gets **exactly those roles** (each id resolved via `RoleManager.FindByIdAsync` *before* the user row is created; one unknown id → `NotFound`, nothing saved); when omitted/empty, the previous default applies — best-effort `RoleNames.User` (`Domain/Constants/RoleNames.cs`) if that role exists yet (checks `RoleExistsAsync` first — if `IdentitySeeder` hasn't run/the DB isn't migrated yet, registration still succeeds, it just skips the role assignment rather than failing). **~~Known security hole~~ closed (2026-07-12)**: `CreateUser` used to be `[AllowAnonymous]`, so honoring caller-supplied `RoleIds` let anyone self-register with any existing role. Resolved by removing self-registration entirely (this is an admin-managed backend): the endpoint now requires the `USER_CREATE` permission, making `RoleIds` admin-gated on create exactly as it already was on update. `Docs/user_role_ids_implementation_guide.md` records the resolution. `UpdateUserCommand` also has `RoleIds` with three-way semantics (null field = leave roles unchanged, `[]` = remove all, non-empty = replace-sync via `AddToRolesAsync`/`RemoveFromRolesAsync`) — that endpoint is permission-gated, so it's not part of the hole. `UserDto` also *returns* `RoleIds` (added 2026-07-10) on every user endpoint — `UserMapper.ToDto(user, roleIds)` takes them as a parameter; `UserService` reads them via a private `GetRoleIdsAsync` (`_dbContext.UserRoles`, one batched query for the whole page in `GetUsersAsync`, not per-row).
- **Deviation from the guide**: the guide's audit-fields section shows registration setting `CreatedBy = user.Email`. This project instead lets the generic `ApplicationDbContext.SaveChangesAsync` interceptor stamp `CreatedBy` from `ICurrentUserService.UserName` — kept for consistency with every other entity in the system rather than special-casing one. Now that creation is admin-gated, `CreatedBy` records the admin who created the account (previously always `"system"`, registration being anonymous).

### Login (`POST /api/auth/login`, `[AllowAnonymous]`)

`AuthService.LoginAsync`, in order:

1. Find user by username. Not found → generic `"Invalid username or password."`
2. `SignInManager.CheckPasswordSignInAsync(user, password, lockoutOnFailure: true)` — locked out → distinct message; wrong password → same generic message as step 1.
3. **Only after the password is confirmed correct**: `IsActive` check, then `EmailConfirmed` check, then (2026-07-12) the per-user IP allowlist check (`IsRequestIpAllowed` — `IsIpRestricted` + `IpAllowlistChecker` against `IHttpContextAccessor`'s `RemoteIpAddress`; also applied in `GoogleLoginAsync` after `IsActive` and in `RefreshTokenAsync`, where it deliberately returns the same generic "invalid token" message), then password-expiry check (`Auth:PasswordExpiryDays`, default 90; days since `LastPasswordChangedTs`; set to `0` to disable) — each with its own specific message.
4. Stamps `LastLoginTs`, issues an access token + refresh token.

**Deviation from the guide**: the guide's login flow lists `IsDeleted`/`IsActive`/`EmailConfirmed` checks *before* password verification. Doing it that way lets an anonymous caller learn account state (exists / deactivated / unverified) from an unauthenticated request just by supplying a username — which directly contradicts the guide's own "never reveal whether email exists" security recommendation. This implementation checks the password *first* and only reveals account-state-specific reasons once the caller has proven they know the correct password. The `IsDeleted` check itself is skipped entirely — `FindByNameAsync` already can't return a soft-deleted user, because of the global `!IsDeleted` query filter (see EF Core entity configuration pattern above), so an explicit check would be dead code.

JWT claims (`Infrastructure/Common/JwtTokenGenerator.cs`): `sub`/`ClaimTypes.NameIdentifier` (user id), `ClaimTypes.Name` (username), `ClaimTypes.Email`, `userType`, `country` (`CountryIso3`), `securityStamp`, `sessionId` (a fresh `Guid` per login/refresh — not persisted or checked against anything yet, it's a placeholder for future per-session revocation), and one `ClaimTypes.Role` per role. **No `permissions` claim** — the guide recommends one, but `AuthorizedAction` already re-checks `Menus`/`RoleClaims` against the database on every request, so embedding a permissions snapshot in the token would just go stale the moment a role's claims change without the token being reissued. Deliberately left out rather than shipping something misleading.

### Refresh tokens, logout, change password

- Refresh tokens are persisted in a **dedicated `RefreshToken` entity** (`Infrastructure/Identity/RefreshToken.cs` + `EntityConfiguration/RefreshTokenConfiguration.cs`, table `identity.refresh_tokens`) — **not** ASP.NET Core Identity's `ApplicationUserToken`/`IdentityUserToken`. This follows the Identity guide's explicit "Enterprise Recommendation": *"Add a dedicated RefreshToken entity for JWT session management, revocation, auditing, multi-device support, and logout handling."* An earlier version of this code abused `ApplicationUserToken`'s custom `RefreshTokenExpiryDate`/`TokenRefreshTokenRevokedDate` fields for this (one row per user, keyed by `LoginProvider`/`Name`) — those fields are now dead (see the identity model table above); the dedicated entity replaced that entirely. **This adds a new table, so it needs a migration** — not created here, same as always.
- **One row per issued token, not one row per user** — this is what makes multi-device sessions actually work: logging in from a second device adds a *second* `RefreshToken` row rather than overwriting the first, so both stay valid independently.
- `POST /api/auth/refresh-token` (`[AllowAnonymous]` — the whole point is the access token has already expired) looks the token up by its `Token` value, checks `IsRevoked`/`IsExpired`, confirms its user is still active, then issues a **new** access + refresh token pair and **rotates**: the presented row is marked `RevokedAtUtc = UtcNow` with `ReplacedByToken` set to the new token's value (an audit trail of the rotation chain), so it can never be redeemed a second time. **Reuse detection (2026-07-12)**: presenting an already-revoked token that is *not* expired is treated as theft of the rotation chain — every active `RefreshToken` for that user is revoked (same `LogoutAsync` path, `CancellationToken.None`) before the generic invalid-token response goes out; a revoked-*and*-expired replay is treated as a stale client and just rejected.
- `POST /api/auth/logout` (authenticated) revokes **every currently-active `RefreshToken` row for that user** — logout-everywhere semantics, since the endpoint only has the caller's user id (from the JWT), not a specific device/token to target. `ChangePasswordAsync`/`ResetPasswordAsync` both call this same method afterward, so changing your password now invalidates every session, not just the one that changed it.
- `POST /api/auth/change-password` (authenticated): validates the new password against the shared `PasswordRules`, requires it differ from the current password, delegates the actual current-password check to `UserManager.ChangePasswordAsync` (which also auto-rotates `SecurityStamp` — no manual `UpdateSecurityStampAsync` call needed, Identity does it internally), stamps `LastPasswordChangedTs`, and revokes all refresh tokens (calls the same logout path) so the response message says "please log in again."

### Email verification & forgot/reset password (SMTP)

- `Application/Common/Interfaces/IEmailService.cs` (`SendEmailAsync(to, subject, htmlBody)`) → `Infrastructure/Email/SmtpEmailService.cs`, a plain `System.Net.Mail.SmtpClient` implementation reading the `Smtp` config section (`Host`/`Port`/`EnableSsl`/`UserName`/`Password`/`FromEmail`/`FromName`) — **all placeholder values in `appsettings.json`, nothing will actually send until real SMTP credentials are supplied.**
- `Infrastructure/Email/EmailLinkBuilder.cs` builds verify/reset links against `App:ClientBaseUrl`. **That's empty by default** (no frontend exists yet) — when empty, the emailed "link" degrades to a raw `userId=...&token=...` query string with instructions implied by context rather than a clickable URL. Set `App:ClientBaseUrl` once a frontend route exists to verify/reset against.
- Registration (`UserService.CreateUserAsync`) automatically sends a verification email. `POST /api/auth/resend-verification-email` (`[AllowAnonymous]`, email in, always the same generic response) re-sends if the account exists and isn't already confirmed.
- `POST /api/auth/verify-email` (`[AllowAnonymous]`, `{ userId, token }`) calls `UserManager.ConfirmEmailAsync`.
- `POST /api/auth/forgot-password` (`[AllowAnonymous]`, email in) **always returns the same generic "if that email exists..." response** regardless of whether the account exists, is active, or is confirmed — internally it only actually generates+sends a reset token when all three are true, but the caller can never distinguish that from "email not found" by the response alone (this is the guide's own "never reveal whether email exists" rule, applied literally here since there's no password check to hide behind first, unlike login).
- `POST /api/auth/reset-password` (`[AllowAnonymous]`, `{ userId, token, newPassword }`) calls `UserManager.ResetPasswordAsync` (also auto-rotates `SecurityStamp`), stamps `LastPasswordChangedTs`, and revokes the refresh token.
- **Email-confirmation and password-reset tokens share one expiry.** `Infrastructure/DependencyInjection.cs` sets `DataProtectionTokenProviderOptions.TokenLifespan = TimeSpan.FromMinutes(30)` globally, because both `GenerateEmailConfirmationTokenAsync` and `GeneratePasswordResetTokenAsync` use the same default `DataProtectorTokenProvider` unless a second named provider is registered. The guide only specifies an expiry for password reset (15–30 min); a 30-minute email-verification link is on the short side for "click whenever you get around to it" — mitigated by the resend endpoint, but if this becomes a real problem, split it by registering a second `DataProtectorTokenProvider<ApplicationUser>` under a different name with its own `TokenLifespan` for email confirmation specifically.

### Google sign-in (`POST /api/auth/google`, `[AllowAnonymous]`)

Verifies a Google-issued **ID token** sent up by the client (`GoogleLoginCommand { IdToken }`) — this is the pattern for a stateless JSON API talking to a browser/mobile client that already has Google's client-side SDK, **not** ASP.NET Core's redirect-based `AddGoogle()`/cookie OAuth middleware. The client (a web frontend using Google Identity Services, or a mobile app using Google Sign-In) gets the ID token directly from Google and POSTs it here; the backend never redirects anywhere and never needs a client secret, only the OAuth **Client ID** (`Authentication:Google:ClientId` in `appsettings.json`, currently a placeholder), used to check the token's audience.

`AuthService.GoogleLoginAsync`:

1. Verifies the token via `GoogleJsonWebSignature.ValidateAsync` (`Google.Apis.Auth` package) — this checks the signature against Google's public keys, expiry, and that `Audience` matches the configured Client ID. An `InvalidJwtException` (bad/expired/wrong-audience token) maps to `ResponseCodes.Unauthorized`, not a 500 — the middleware would otherwise turn it into a generic server error, which is the wrong signal for "this token is bad."
2. Looks up an existing login via `UserManager.FindByLoginAsync("Google", payload.Subject)` — keyed on Google's `sub` claim (`payload.Subject`), **not** email, since a Google account's email can change but its `sub` can't. **This is the one external-login case where using `UserManager`'s own APIs (`FindByLoginAsync`/`AddLoginAsync`) is correct and safe** — unlike the `RoleClaim`/`RefreshToken` situations above, `ApplicationUserLogin` has no custom columns beyond the `IAuditableEntity` fields (which the `DbContext` interceptor stamps automatically), so there's nothing for the manager API to silently miss.
3. If no existing Google login: looks up by email. An existing local account (e.g. registered with a password) gets the Google login **linked** to it via `AddLoginAsync` — same account, now two ways to sign in. If no account exists at all, creates a new `ApplicationUser` — `UserType.User` (same lockdown as normal registration), `EmailConfirmed = payload.EmailVerified` (Google already verified it), and **no password** (`CreateAsync(user)`, no password argument) — the account can only ever sign in via Google until the user adds a password via `POST /api/auth/set-password`. Username is derived from the email's local part, de-duplicated with a numeric suffix if taken (`GenerateUniqueUserNameAsync`). A brand-new Google-created account also gets the best-effort default `RoleNames.User` role (private `AssignDefaultRoleAsync`, same shape as `UserService`'s) — added 2026-07-10 because previously these accounts had zero roles and an empty menu tree, unlike password registration.
4. `IsActive` is checked the same as normal login (deactivated accounts can't sign in via Google either); `LastLoginTs` is stamped; tokens are issued through the same `IssueTokensAsync` helper `LoginAsync`/`RefreshTokenAsync` use, so a Google sign-in gets a real access + refresh token pair like any other login.
5. Deliberately **skips** the password-expiry check from normal login — meaningless for a passwordless account, and `LastPasswordChangedTs` is `null` for one anyway (the expiry check already no-ops on `null`).

### SuperAdmin protections

`UserService`: `UpdateUserAsync` refuses to set `IsActive = false` on a `UserType.SuperAdmin` user (`ResponseCodes.Forbidden`); `DeleteUserAsync` refuses to delete one at all. Both guide rules ("SuperAdmin cannot be deleted", "SuperAdmin cannot be deactivated") implemented as simple type checks — there's exactly one `UserType.SuperAdmin` tier, so "the current SuperAdmin can't lock out all other SuperAdmins" isn't a distinct scenario to guard against yet.

### Role-claim assignment (menu permissions) & user-role lookup

This is the piece that actually lets `AuthorizedAction` grant anyone access — `Menus`/roles/users existing on their own doesn't connect them; a role needs `ApplicationRoleClaim` rows pointing at `Menu`s, and a user needs an `ApplicationUserRole` row pointing at a role that has those claims.

- `GET /api/roles/user-menus` (`RolesController.GetUserRoles`, → `IRoleService.GetUserRolesAsync`) — returns the **current** user's permitted menu tree (`CommonResponse<List<MenuClaimDto>>`, roots only, children nested): resolves the caller from `ICurrentUserService.UserId` (no `userId` parameter), gets their role names via `UserManager.GetRolesAsync`, translates to role ids, collects the distinct `MenuId`s from those roles' `ApplicationRoleClaim` rows, then walks each granted menu's ancestor chain so `SUB_MENU`/`MAIN_MENU` parents are included, and assembles the tree in memory (`Children` populated, `HasChildren` computed, ordered by `Order`). This is the post-login nav endpoint, so it's listed in `DefaultEnabledMenu` (`"Roles": "GetUserRoles"`) — any authenticated user can call it without a permission row; zero grants returns an empty list, not a 403. The action kept the name `GetUserRoles` so the seeded `ROLE_USER_ROLES` permission row still matches. `MenuClaimMapper` lives in `Application/Roles/` (not `Infrastructure/Identity/Mapper/`) because it maps the Domain `Menu` entity, not an Infrastructure-only type.
- `POST /api/roles/claims` (`AssignMenuToRoleAsync`, body `{ roleId, menuId }`) and `DELETE /api/roles/{roleId}/claims/{menuId}` (`RemoveMenuFromRoleAsync`) manage `ApplicationRoleClaim` rows directly; `GET /api/roles/{roleId}/claims` (`GetRoleClaimsAsync`) reads a role's existing grants back as `List<RoleClaimDto>` (404 if the role doesn't exist; `Include(rc => rc.Menu)` so the mapper can fill `MenuCode`/`MenuDisplayName`).
- **`RoleService` (`Infrastructure/Identity/Services/RoleService.cs`) injects `ApplicationDbContext` directly, alongside `RoleManager`/`UserManager`.** `RoleManager<TRole>.AddClaimAsync(role, new Claim(type, value))` exists but only sets the base `IdentityRoleClaim` shape (`ClaimType`/`ClaimValue`) — it has no knowledge of our custom `MenuId` column, so using it here would silently leave `MenuId` at `0` and break every permission check built on it. The same class of problem showed up twice more since: `ApplicationUserToken`'s custom refresh-token fields (now moot — replaced by the dedicated `RefreshToken` entity) and would show up again for any future custom column bolted onto a base Identity type. Same fix every time: bypass the manager/store, write the row directly through `ApplicationDbContext`. Contrast this with Google sign-in below, where `UserManager`'s own login APIs *are* safe to use, precisely because `ApplicationUserLogin` has no such custom columns.
- `AssignMenuToRoleAsync` validates the command (`RoleId`/`MenuId` non-empty via `AssignMenuToRoleCommandValidator`), confirms both the role and the menu exist (`ResponseCodes.NotFound`), rejects a duplicate `(RoleId, MenuId)` pair (`ResponseCodes.Conflict`), then inserts a new `ApplicationRoleClaim` with `ClaimType = "Permission"` and `ClaimValue = menu.Code` — these two columns aren't read by `AuthorizedAction` (it only looks at `MenuId`) but are set anyway for readability if anyone inspects the table directly. Both `Menu` and `ApplicationRole` navigation properties are set explicitly on the new row (not left for EF's relationship-fixup to figure out) so `RoleClaimMapper.ToDto` can read `roleClaim.Menu.Code`/`DisplayName` immediately, before a re-query.
- `RoleClaimMapper.cs` lives in `Infrastructure/Identity/Mapper/` (alongside `UserMapper.cs`/`RoleMapper.cs`), not `Application/Roles/`, because it maps `ApplicationRoleClaim` — an Infrastructure-only type. `RoleClaimDto` itself lives in `Application/Roles/Dtos/`, same split as everywhere else.
- `POST /api/roles/users` (`AssignRoleToUserAsync`, body `{ userId, roleId }`) and `DELETE /api/roles/users/{userId}/{roleId}` (`RemoveRoleFromUserAsync`) manage `ApplicationUserRole` rows — via `UserManager.AddToRoleAsync`/`RemoveFromRoleAsync`, which **are** safe here (unlike role-claims): `ApplicationUserRole`'s only custom columns are the `IAuditableEntity` fields, stamped automatically by the `DbContext` — same reasoning as Google login's `AddLoginAsync`. Assign 409s if the user is already in the role; remove 404s if they're not.

### Identity seeding (`Infrastructure/Persistence/DataSeeder/IdentitySeeder.cs`)

`IdentitySeeder.SeedAsync(IServiceProvider)` creates the `SuperAdmin`/`Admin`/`User` roles (names from `Domain/Constants/RoleNames.cs` — the same constants `UserService.CreateUserAsync`'s default-role assignment uses, so a role renamed in one place doesn't silently drift from the other) and one `ApplicationUser` per role (assigned via `UserManager.AddToRoleAsync`), reading username/email/password/first+last name from the `Seed:SuperAdmin`/`Seed:Admin`/`Seed:User` sections of `appsettings.json`. **Idempotent** — checks `RoleExistsAsync`/`FindByNameAsync` first and skips anything that already exists, so it's safe to run on every startup rather than needing a one-off CLI command.

- Called from `WebApi/Program.cs` right after `app.Build()`, wrapped in try/catch that logs a warning and lets the app start anyway on failure — **this is what keeps a not-yet-migrated database from crashing startup.** Since migrations aren't created in this repo (owned by the user), seeding will fail every run until migrations exist and have been applied; once they have, the very next startup seeds successfully and every subsequent one is a no-op.
- Seeded users get `EmailConfirmed = true` and `IsActive = true` directly (bypassing the normal registration/verification flow entirely, since these are bootstrap system accounts, not self-registered users) — they can log in immediately.
- Goes through `UserManager`/`RoleManager` directly, **not** `IUserService.CreateUserAsync` — that method hardcodes `UserType.User` (deliberately, for the public registration endpoint) and would make seeding an Admin/SuperAdmin impossible. Bootstrapping identity data is treated as an infrastructure concern, same as `AuthorizedAction` reaching into `ApplicationDbContext` directly.
- `Seed:*:Password` values in `appsettings.json` are dev-only credentials — **treat them exactly like the `Jwt:Key`/`Smtp:*` placeholders: don't reuse them anywhere beyond local dev.**

### Menu & permission seeding (`Infrastructure/Persistence/DataSeeder/MenuSeeder.cs`)

Runs right after `IdentitySeeder` in `Program.cs` (same try/catch, same "skip until migrated" behavior). **All seed data is defined in code, not configuration** — deliberately: the menu/permission catalog is structural (it must exactly mirror the controllers/actions that exist in this codebase), so it belongs under version control next to the code it describes, unlike per-environment credentials which stay in `appsettings.json`. Idempotent by `Menu.Code` / `(RoleId, MenuId)` — safe on every startup; new permissions added to the seeder later get picked up on the next boot without touching existing rows.

Seeds, in order:

1. Three `MAIN_MENU` rows (`USER_MANAGEMENT`, `ROLE_MANAGEMENT`, `MENU_MANAGEMENT`), `MenuFor = ADMIN`, visible.
2. One `PERMISSION` leaf per protected endpoint (e.g. `USER_LIST` → `Users`/`GetUsers`, `ROLE_CLAIM_ASSIGN` → `Roles`/`AssignMenuToRole`), parented to its main menu, `IsHidden = true` (permissions are authorization records, not nav items).
3. **Every `PERMISSION` menu granted to the SuperAdmin role** as `ApplicationRoleClaim` rows — this is what makes the seeded superadmin usable end-to-end from first boot. `Admin`/`User` roles deliberately get nothing; granting them is a runtime decision made through `POST /api/roles/claims` while signed in as superadmin.

When adding a new controller/action that should be permission-gated: add its `PERMISSION` row (and main menu, if it's a new area) to `MenuSeeder` — otherwise nobody, superadmin included, can call it, and remember `Menu.Controller` must be the route value (`"Users"` for `UsersController`), not the class name.

## System configs (dropdowns), app settings, access log, error log

Four cross-cutting subsystems, all built as plain Domain entities + `IUnitOfWork` repositories (so their services live in `Application`, per the rule of thumb above). **All five tables (`dbo.config_types`, `dbo.configs`, `dbo.app_configs`, `dbo.system_access_logs`, `dbo.error_logs`) need a migration** — user-owned, as always.

### Config catalog (`Application/Configs/`, `ConfigsController`)

**Every dropdown in the system is sourced from `ConfigType`/`Config`**, not hardcoded lists: a `ConfigType` names a dropdown and owns an `int TypeCode`; its `Config` rows are the options (`Code`/`Label`/`Order` + three free-form `AdditionalValue` columns). The `Config.TypeCode → ConfigType.TypeCode` FK targets an **alternate key** (not the Guid PK) — `ConfigConfiguration` sets `HasPrincipalKey` explicitly, same shadow-FK caution as `RefreshToken`. `(TypeCode, Code)` is unique. Endpoints: `POST/GET /api/configs/types` (create/list types), `PUT/DELETE /api/configs/types/{id}` (update/delete type — `TypeCode` is immutable on update since it's the alternate key options FK onto; delete is refused with `Conflict` while the type still has `Config` rows, pre-checked via `IConfigRepository.AnyByTypeCodeAsync` because the `Restrict` FK would otherwise turn it into a 500), `POST /api/configs` (create option), `GET /api/configs/types/{id}` / `GET /api/configs/{id}` (single-item lookups, `ConfigTypeDto`/`ConfigDto`), `PUT/DELETE /api/configs/{id}` (update/delete option — update keeps `TypeCode` immutable and re-checks `Code` uniqueness only when it changes, mirroring `UpdateRoleCommand`; delete is **hard**, `Config` is only `AuditableEntity`), `GET /api/configs/dropdown/{typeCode}` (options ordered by `Order` — listed in `DefaultEnabledMenu`, so any authenticated user can populate dropdowns without a permission grant). **The dropdown endpoint returns `DropdownItemDto` (`Application/Common/Models/`) — `Value`/`Label`/`Order`/`AdditionalValue1..3` — not `ConfigDto`**: it's the common dropdown shape, deliberately placed in `Common/Models` so every future dropdown endpoint returns the same type and one frontend select component binds them all. `ConfigDto` (with `Id`/`TypeCode`/`Code`) stays the admin-CRUD shape.

### App settings store (`Application/AppConfigs/`, `AppConfigsController`)

`AppConfig` (`dbo.app_configs`) is the **application-wide settings key/value store** — app name, theme mode, primary/secondary colors, logo URL, etc. — distinct from the `ConfigType`/`Config` dropdown catalog: flat rows (`ConfigParam` unique across the table via `ix_app_configs_config_param`, `ConfigValue` ≤ 555, free-form **optional** `ConfigGroup` — made nullable 2026-07-10; the original `app_configs` migration created the column `NOT NULL`, so a follow-up ALTER migration is needed before a null group actually saves — `IsEnable`), no type hierarchy. Standard admin CRUD at `/api/appconfigs` (create pre-checks `IAppConfigRepository.ConfigParamExistsAsync` → `Conflict`; update re-checks only when `ConfigParam` changes; delete is **hard**, `AppConfig` is only `AuditableEntity`; group lookup `GET /api/appconfigs/group/{configGroup}`, exact-match, unpaged). The one special endpoint: **`GET /api/appconfigs/public` is `[AllowAnonymous]`** — the UI bootstrap call (the login page needs the app name/colors before any token exists) — returning only `IsEnable = true` rows as the trimmed `PublicAppConfigDto` (`ConfigParam`/`ConfigValue`/`ConfigGroup`, no ids), so **an enabled app config is world-readable: never store secrets in one**; disabling a row (`IsEnable = false`) hides it from the public endpoint while keeping it in the admin CRUD. Permission rows (`APP_CONFIG_*`) seed under `CONFIG_MANAGEMENT` in `MenuSeeder` with `Controller = "AppConfigs"`. `AppConfigSeeder` (2026-07-12, runs after `MenuSeeder` in `Program.cs`) seeds a 17-row baseline (`GENERAL`/`THEME`/`ANNOUNCEMENT` groups) — idempotent by `ConfigParam` and strictly **create-if-missing**: it never updates an existing row, so admin edits survive restarts. The full seeded catalog + further suggested params live in `Docs/app_config_implementation_guide.md`, which is this feature's UI contract alongside its summary section in `UI-Implementation-Guide.md` — keep both in sync on endpoint changes.

### System access log (`Application/AccessLogs/`, written by `AuthorizedAction`)

`SystemAccessLog` records who called what (`UserId`/`UserName`/`Controller`/`Action`/`HttpMethod`/`Url`/`IpAddress`; timestamp via audit stamping). **Only** controller/action pairs listed in the `CriticalChanges` appsettings section are logged (configurable — same `{ "Controller": "Action1,Action2" }` shape as `DefaultEnabledMenu`), and only for requests that passed authorization. Written with `CancellationToken.None` (a disconnect must not skip the audit row) but wrapped in try/catch (a logging failure must not fail the request). Read back via `GET /api/dashboard/access-logs` (paged, `CreatedTs` desc, optional `userId` query filter).

### Error log (`Application/ErrorLogs/`, written by `ExceptionHandlingMiddleware`, read via `DashboardController`)

`ErrorLog` rows are **deduplicated, not appended**: `ErrorLogService.LogErrorAsync` fingerprints each error (SHA-256 over exception type + message + stack trace, stored in an indexed `fingerprint_hash` column) and increments `ErrorCount`/`LastOccurredTs` on an existing row instead of inserting a duplicate. The fingerprint index is deliberately **not unique** — a race on a brand-new error may produce a duplicate row, which is preferable to a unique-violation exception inside the error-logging path. The middleware persists through a **fresh DI scope** (`IServiceScopeFactory`), not `RequestServices` — the request's scoped `DbContext` may be the very thing that failed — with `CancellationToken.None`, inside its own try/catch so a down database doesn't turn error logging into a second failure. Client-cancellation (`OperationCanceledException`) is not persisted, matching its info-level treatment. Read endpoints: `GET /api/dashboard/error-logs` (paged, `LastOccurredTs` desc) and `GET /api/dashboard/error-logs/summary` (distinct vs. total counts). `GET /api/dashboard/summary` adds user counts (total + active, soft-deleted excluded by the global filter) to those error counts — its `DashboardService` lives in `Infrastructure/Identity/Services/` (not Application) because counting users needs the Identity `Users` set; the contract `IDashboardService` sits in `Application/Dashboard/`.

## Student management (2026-07-12, restructured 2026-07-13)

A full sub-system built from `Docs`-external design input (`Student_Management_System_Database_Design.md`), adapted to this codebase's conventions. Twelve Domain entities (`AcademicYear`, `AcademicClass`, `ClassSection`, `ClassSubject`, `Teacher`, `TeacherQualification`, `TeacherAssignment`, `Guardian`, `Student`, `StudentGuardian`, `Enrollment`, `EnrollmentSubject`), six Application features (`AcademicYears`, `AcademicClasses`, `Teachers`, `Guardians`, `Students`, `Enrollments` — all `IUnitOfWork`-based, so services live in Application), six controllers, six aggregate repositories (each owns its child link entities — e.g. `ITeacherRepository` also handles qualifications/assignments and `IAcademicClassRepository` owns sections + subjects; don't add separate repos for pure link tables). UI contract: `Docs/student_management_implementation_guide.md` + `Docs/class_section_structure_implementation_guide.md` (the 2026-07-13 restructure) + the summary section in `UI-Implementation-Guide.md`.

**2026-07-13 restructure** (breaking, needs a user-owned migration — see Known gaps):

- **`AcademicClass` is now one grade per year** (`(AcademicYearId, GradeCode)` unique); the old per-section class rows split into the child **`ClassSection`** entity (`SectionCode`, `Capacity`, `Status`; `(AcademicClassId, SectionCode)` unique; soft-deleted). `AcademicClassDto` nests `Sections`; class create accepts optional inline sections; class delete refuses while sections exist; section delete refuses while enrollments exist. New `CLASS_SECTION_*` permission rows in `MenuSeeder`.
- **`ClassSubject` still hangs off `AcademicClass`** — one subject list per grade, shared by every section (the "same class, same subjects" rule) — but it also carries a nullable `ClassSectionId` (added same day): null = class-wide; set = an **optional** subject offered only in that section (mandatory + `ClassSectionId` is a `ValidationError`). Invariant: a subject appears either once class-wide or once per section, never both — enforced in `AssignSubjectAsync` via `GetClassSubjectRowsByCodeAsync` (the unique index `(class, subject, section)` can't block duplicate NULL rows, same Postgres NULLs-distinct caveat as teacher assignments). A section-scoped subject is only electable by students enrolled in that section and only teachable via assignments targeting that section (or none); section delete also refuses while scoped subjects point at it. `GET /api/academicclasses/{id}/subjects?classSectionId=…` returns one section's *effective* list (class-wide + scoped rows).
- **Per-year subject rows are deliberate, not duplication** — curricula/teachers/electives are per-year facts, so year-less "global" grade→subject mappings were rejected (they'd rewrite history on every curriculum edit). The re-entry cost is solved by **`POST /api/academicyears/{id}/clone-structure`** (`AcademicYearService.CloneStructureAsync`, body `{ sourceAcademicYearId }`): copies the source year's classes → sections → subject mappings (section-scoped rows remapped onto the cloned sections by `SectionCode`) into the target year in one `SaveChangesAsync`; grades already present in the target are skipped and reported (`CloneStructureResultDto.SkippedGradeCodes`); enrollments/teacher assignments are never copied. Seeded permission: `YEAR_CLONE_STRUCTURE`.
- **`Enrollment.AcademicClassId` → `ClassSectionId`** — students enroll into a section. Capacity and roll-number uniqueness are per section. **New invariant: one `Enrolled`-status row per student per academic year** (`IEnrollmentRepository.HasActiveEnrollmentInYearAsync`, checked on create and when an update flips status back to `Enrolled`) — fixes the same-student-in-two-grades bug.
- **Guardians at onboarding**: `CreateStudentCommand.Guardians` (list of `StudentGuardianInput` — existing `GuardianId` *or* inline new-guardian fields, plus `RelationshipCode`/`IsPrimary`, at most one primary, all resolved before anything is created so the save is all-or-nothing in one `SaveChangesAsync`). `StudentDto.Guardians` is filled on detail/create (empty on the paged list).
- **Teacher documents** (2026-07-13): `TeacherDocument` (`dbo.teacher_documents`, hard-deleted, owned by `ITeacherRepository`) stores uploaded verification files — `DocumentTypeCode` from the new Config catalog **1006** (`ConfigTypeCodes.DocumentType`, options seeded by `ConfigCatalogSeeder`), `DocumentName`, original `FileName`, storage-relative `FilePath` (never exposed in DTOs), `ContentType`, `FileSizeBytes`, optional `ValidUntil` expiry (license/police-report style docs), `Remarks`. Upload is **multipart/form-data** on `POST /api/teachers/{id}/documents` — the controller unwraps `IFormFile` into primitives (Application can't see ASP.NET Core types) and `TeacherService.UploadDocumentAsync` enforces the rules: extensions/size live in `Application/Common/Validation/DocumentFileRules.cs` (.pdf/.jpg/.jpeg/.png, 10 MB — code constants, deliberately not config). Files are stored via `IFileStorageService` (`Application/Common/Interfaces`) → `Infrastructure/Files/LocalFileStorageService` (singleton; root from `FileStorage:RootPath`, default `Uploads` under content root; GUID file names; path-traversal guard; `Delete` is best-effort). Failed row-save deletes the just-stored file; document delete removes the row first, then the file. Download (`GET …/download`) streams the raw file — the one non-enveloped success response. Permissions: `TEACHER_DOCUMENT_UPLOAD/LIST/DOWNLOAD/DELETE`. Guide: `Docs/teacher_documents_implementation_guide.md`.
- **Student profile & update enhancements** (2026-07-13, `Docs/student_profile_enhancements_implementation_guide.md`): `GET /api/students/{id}` adds `CurrentEnrollment` (active enrollment preferring the `IsCurrent` year, else latest by `StartDate` — `IEnrollmentRepository.GetActiveByStudentAsync`; subjects = the section's mandatory rows + chosen electives). `PUT /api/students/{id}` takes `Guardians` with **three-way semantics** mirroring `UpdateUserCommand.RoleIds` (null = untouched, `[]` = unlink all, list = replace-sync via `StudentService.SyncGuardianLinksAsync` — existing links update relationship/primary in place, inline entries create new guardians; everything validated before the first mutation, one `SaveChangesAsync`).
- **Teacher assignments are section-aware**: `TeacherAssignment.ClassSectionId` is nullable (null = teaches the subject to every section of the class; when set, the section must belong to the `ClassSubject`'s class). Uniqueness is per (teacher, classSubject, classSection) — service-checked via `AssignmentExistsAsync`, because the DB unique index can't catch duplicate NULL-section rows (Postgres NULLs-are-distinct). `IsClassTeacher = true` **requires** a `ClassSectionId`, and at most one class-teacher assignment exists per section (`ClassTeacherExistsForSectionAsync`). Section delete also refuses while assignments target it.

Deliberate design decisions (deviations from the raw design doc):

- **Grade/Section/Subject/GuardianRelationship/TeacherQualification are NOT tables** — they are Config catalogs (`Domain/Constants/ConfigTypeCodes.cs`: 1001–1005, types seeded by `ConfigCatalogSeeder`, which also seeds relationship + qualification options). Entities store the option's **`Code` string** (`GradeCode`, `SubjectCode`, `RelationshipCode`, …) validated in services via `IUnitOfWork.Configs.CodeExistsAsync(typeCode, code)` — **not database FKs**, because `Config.Code` is only unique per `(TypeCode, Code)`. Corollary: deleting a referenced Config option orphans the code strings pointing at it; nothing enforces that today.
- **`TeacherQualification` is a table the design doc lacked** — the doc had a single `Qualification VARCHAR` on Teacher; real qualification records (code + course + institution + year + score) needed their own rows.
- **No student/teacher logins** — records only, no `UserId` linkage. When logins come, the linkage entity must live in `Infrastructure/Identity/` (references `ApplicationUser`), per the `RefreshToken` placement rule.
- Soft vs hard delete: main records (`AcademicYear`, `AcademicClass`, `Teacher`, `Guardian`, `Student`, `Enrollment`) are `SoftDeleteAuditableEntity`; pure link/child rows (`ClassSubject`, `TeacherQualification`, `TeacherAssignment`, `StudentGuardian`, `EnrollmentSubject`) are `AuditableEntity` (hard). All uniqueness pre-checks use `IgnoreQueryFilters()` (soft-deleted rows keep codes/pairs reserved → clean 409).
- Soft vs hard delete addendum (2026-07-13): `ClassSection` is also `SoftDeleteAuditableEntity` (its `(class, section)` pair stays reserved like the other identity-bearing records).
- Service-enforced invariants: one `IsCurrent` academic year (setting it demotes others), one `IsClassTeacher` per **section** (and a class-teacher assignment must name a section), one `IsPrimary` guardian per student (auto-demote on link; validated to at most one in the onboarding list), **one active (`Enrolled`) enrollment per student per academic year**, section capacity (`0` = unlimited; only `Enrolled`-status rows count), per-section roll-number uniqueness, electives must be non-mandatory subjects of the enrollment's own class (reached through its section), deletes refused with `Conflict` while children exist (year→classes, class→sections, section→enrollments/teacher-assignments, teacher→assignments, guardian→links, classSubject→assignments/electives). Class identity (year/grade), section identity (`SectionCode`), `AcademicYear.Code`, `Teacher.EmployeeNo`, `Student.AdmissionNo`, and an enrollment's student/section are immutable on update.

## Known gaps (as of writing)

- Migrations are **user-created** — don't run `dotnet ef` commands. Migrations for the initial schema, the config/log tables, `app_configs`, the `config_group` nullable ALTER, the `is_ip_restricted`/`user_ip_allowed` columns, and the original 11 student-management tables all exist under `Infrastructure/Migrations/`. **Pending as of 2026-07-13**: the class/section restructure needs a migration (new `dbo.class_sections`; `academic_classes` loses `section_code`/`capacity`, unique index becomes `(year, grade)`; `enrollments.academic_class_id` → `class_section_id` with new indexes; `teacher_assignments` gains nullable `class_section_id`, unique index becomes `(teacher_id, class_subject_id, class_section_id)`; `class_subjects` gains nullable `class_section_id`, unique index becomes `(academic_class_id, subject_code, class_section_id)`; new `dbo.teacher_documents`) — plus a data decision for any existing rows (each old grade+section class becomes one class + one section). Note the DB can't create a unique index over rows that already violate it — on a dev DB, clearing the class/enrollment tables before `database update` is the pragmatic path (see 2026-07-13 conversation).
- Neither log table (`system_access_logs`, `error_logs`) has retention/pruning — same "later problem" status as `RefreshToken` rows.
- ~~`IsIpRestricted`/`UserIpAllowed` are stored but not enforced~~ **resolved (2026-07-12)**: enforced both at token issuance (login/Google/refresh in `AuthService`) and per-request in `AuthorizedAction`, via `Infrastructure/Common/IpAllowlistChecker`. Remaining flavor of the gap: `ForwardedHeadersMiddleware` is **not** configured, so behind a reverse proxy `RemoteIpAddress` is the proxy — configure it (with `KnownProxies`) before relying on the allowlist (or the rate limiter's per-IP partitioning) in such a deployment.
- ~~Chicken-and-egg on first permissions~~ **resolved**: `MenuSeeder` now seeds the menu/permission catalog in code and grants everything to SuperAdmin at startup. Remaining flavor of the gap: `Admin` and `User` roles start with zero permissions until someone grants them via the API, and any *new* endpoint is uncallable until its `PERMISSION` row is added to `MenuSeeder` (or created via `POST /api/menus`).
- `Jwt:Key`, every `Smtp:*` credential, and `Authentication:Google:ClientId` in `appsettings.json` are still literal placeholders.
- ~~Google-created accounts have no flow to set a password~~ **resolved (2026-07-10)**: `POST /api/auth/set-password` (authenticated, listed in `DefaultEnabledMenu` under `"Auth"`) lets a passwordless account add one via `UserManager.AddPasswordAsync` — refused with `Conflict` if the account already has a password (those must use `change-password`, which proves knowledge of the current one). Deliberately does **not** revoke sessions: nothing pre-existing was invalidated. See `Docs/set_password_implementation_guide.md`.
- `RefreshToken` rows are never pruned — expired/revoked rows accumulate forever. Fine for now; a cleanup job (or just filtering `IsActive` in queries, which is already done) is a later problem, not a correctness one.
- `IIdentityService.CreateUserAsync` (the older, lower-level primitive in `Application/Common/Interfaces/`) now takes `firstName`/`lastName` (fixed 2026-07-12 — creates through it no longer violate the `NOT NULL` name columns) but still doesn't cover roles/ToS/profile fields — it's intentionally separate from `IUserService.CreateUserAsync` (the full-featured one behind `UsersController`). Note the whole interface currently has **no callers** — it's kept as a template primitive; extend or drop it deliberately.
- `Application` has `Menus`, `Users`, `Roles`, `Auth` so far — everything else in the domain (posts, categories, etc.) still needs its own feature folder following the pattern above.
- ~~User↔role assignment~~ **resolved**: `POST /api/roles/users` / `DELETE /api/roles/users/{userId}/{roleId}` create/remove `ApplicationUserRole` rows (see Role-claim assignment section). Note there's no guard against removing the last role from (or any role of) a SuperAdmin — the SuperAdmin *bypass* keys off `UserType`, not roles, so they keep working regardless.
- **Deleting a user is soft, deleting a role is hard.** `DELETE /api/users/{id}` → `ApplicationDbContext`'s soft-delete interceptor converts it to `IsDeleted = true` (since `ApplicationUser` is `ISoftDeleteAuditableEntity`). `DELETE /api/roles/{id}` actually removes the row (`ApplicationRole` only implements `IAuditableEntity`). Intentional, matches the interfaces each entity implements — but easy to assume both behave the same way.
- `UpdateUserCommand` deliberately excludes `UserName`/`Email`/`Password` (those need dedicated flows with uniqueness re-checks — `Email`/`UserName` changes aren't implemented at all yet; `Password` goes through `change-password`/`reset-password`). `UpdateRoleCommand` *does* allow renaming `Name`, re-checking `RoleExistsAsync` only when the name actually changes.
- Only Google is wired up for external login — the guide mentions "Google, Microsoft, Apple, or GitHub" generically. `ApplicationUserLogin`/`UserManager.FindByLoginAsync`/`AddLoginAsync` are provider-agnostic, so adding another provider means: add its package, verify its token format (most have an equivalent "validate this token, get back a payload with a stable subject id" library), and follow the same shape as `GoogleLoginAsync` — no schema changes needed.
- From the guide's "Security Recommendations" checklist, **not implemented**: MFA, session management (`sessionId` claim exists but nothing tracks/revokes individual sessions), audit logging of auth events specifically (general audit stamping exists, but not a security event log). Implemented: lockout, email verification, SecurityStamp rotation on password change, refresh-token revocation on password change, short-lived access tokens, no-enumeration responses, per-IP rate limiting on the auth endpoints (2026-07-12, see the rate limiting bullet above + `Docs/rate_limiting_implementation_guide.md`), per-user IP allowlists (2026-07-12), refresh-token reuse detection + CORS origin allowlist + security response headers (2026-07-12, `Docs/security_hardening_implementation_guide.md`). Note anti-CSRF is deliberately **not** needed: auth is pure `Authorization: Bearer` (no cookies), so browsers never auto-attach credentials — revisit only if a token ever moves into a cookie.
